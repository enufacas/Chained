# Repository Organization Guidelines

> **Created by @organize-guru** following clean code principles

## Purpose
This file defines rules for maintaining a clean and organized repository structure. Following these guidelines prevents clutter and ensures maintainability.

## Root Directory Policy

### ✅ Files That SHOULD Be in Root

**Core Documentation Only:**
- `README.md` - Main project overview
- `LICENSE` - Project license
- `FAQ.md` - Frequently asked questions
- `GETTING_STARTED.md` - Initial setup guide
- `QUICKSTART.md` - Quick reference guide
- `CONFIGURATION.md` - Configuration reference
- `AGENT_QUICKSTART.md` - Agent system quick start
- `AUTONOMOUS_SYSTEM_ARCHITECTURE.md` - Core architecture document
- `AUTONOMOUS_LOOP_IMPLEMENTATION.md` - Implementation guide
- `SYSTEM_RECREATION_GUIDE.md` - System recreation documentation
- `SYSTEM_RECREATION_PROMPT.md` - Recreation prompt

**Build/Config Files:**
- `.gitignore`
- `.github/` directory (workflows, actions, etc.)
- `requirements.txt`
- Configuration files specific to build tools

### ❌ Files That Should NOT Be in Root

**Implementation Summaries** → `docs/implementation-summaries/`
- Agent-related: `docs/implementation-summaries/agents/`
- Workflow-related: `docs/implementation-summaries/workflows/`
- Feature-related: `docs/implementation-summaries/features/`
- Infrastructure: `docs/implementation-summaries/infrastructure/`

**Investigation Reports** → `docs/investigations/`
- Analysis reports
- Bug investigation reports
- Health check reports
- Performance analysis

**Guides and Tutorials** → `docs/guides/`
- How-to guides
- Usage guides
- Validation guides
- Quickstart guides (except core ones)

**Architecture Documents** → `docs/architecture/`
- System design documents
- Architecture diagrams
- Technical specifications
- Component designs

**Test Files** → `tests/`
- All `test_*.py` files
- Test scripts
- Test data fixtures

**Utility Scripts** → `tools/` or `scripts/`
- Analysis scripts
- Data processing scripts
- Validation scripts
- One-off utilities

**Migration Scripts** → `scripts/migrations/`
- Database migrations
- Data migrations
- One-time migration utilities

## Naming Conventions

### File Naming
- Use `SCREAMING_SNAKE_CASE` for high-level documentation: `GETTING_STARTED.md`
- Use `kebab-case` for implementation details: `feature-implementation.md`
- Use descriptive names that indicate content: `agent-evaluation-summary.md` not `summary.md`

### Directory Naming
- Use `kebab-case` for directory names: `implementation-summaries/`
- Plural for collections: `guides/`, `investigations/`, `tests/`
- Clear hierarchy: `docs/implementation-summaries/agents/`

## When Creating New Files

### Decision Tree

1. **Is this core project documentation?**
   - YES → Root directory
   - NO → Continue to step 2

2. **Is this an implementation summary or completion report?**
   - YES → `docs/implementation-summaries/[category]/`
   - NO → Continue to step 3

3. **Is this an investigation or analysis report?**
   - YES → `docs/investigations/`
   - NO → Continue to step 4

4. **Is this a user guide or tutorial?**
   - YES → `docs/guides/` or `docs/tutorials/`
   - NO → Continue to step 5

5. **Is this architecture documentation?**
   - YES → `docs/architecture/`
   - NO → Continue to step 6

6. **Is this a test file?**
   - YES → `tests/`
   - NO → Continue to step 7

7. **Is this a utility script?**
   - YES → `tools/` or `scripts/`
   - NO → Consider if file is needed

## Enforcement

### For All Contributors
- Before creating files in root, check this guide
- PR reviewers should verify proper file placement
- Automated checks may reject PRs with improper file placement

### For Autonomous Agents
- **@organize-guru** periodically reviews and organizes
- Agents should follow this guide when creating documentation
- Issue #[number] tracks root directory cleanup

## Rationale (SOLID Principles)

Following **@organize-guru**'s clean code philosophy:

1. **Single Responsibility**: Each directory has one clear purpose
2. **Open/Closed**: Easy to add new docs without restructuring
3. **Liskov Substitution**: Files in same category are interchangeable
4. **Interface Segregation**: Directory structure segregates concerns
5. **Dependency Inversion**: High-level docs in root, details in subdirs
